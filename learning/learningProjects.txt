### Phase 1: The Foundation - Understanding REST & Web Architecture
#### 1.1 What is REST?
REST (Representational State Transfer) is an architectural style for providing standards between computer systems on the web, making it easier for systems to communicate with each other.

- **Why it's needed:** Without a standard, every API would have its own custom rules for how data is requested and sent, leading to massive integration headaches.
- **How it works:** It uses standard HTTP methods (GET, POST, PUT, DELETE) to perform CRUD (Create, Read, Update, Delete) operations on resources identified by URLs.

#### 1.2 HTTP Methods & Common Pitfalls
- **GET:** Retrieve data. Should never change server state.
- **POST:** Create data.
- **PATCH/PUT:** Update data.
- **DELETE:** Remove data.

**The 405 Method Not Allowed Error:**
- **What is it?** This happens when the client sends an HTTP request using a method that the server doesn't support for that specific URL.
- **Why it happens in LearningHub:** If a view is set as `ReadOnlyModelViewSet`, it only supports GET. If a test tries to POST to it, you get a 405.
- **How to fix:** Ensure the Viewset supports the required actions (e.g., using `ModelViewSet` instead of `ReadOnlyModelViewSet`) or adjust the request method.

### Phase 2: Backend Logic & Data Integrity
#### 2.1 Database Constraints
- **Unique Constraints:** Ensures no two rows have the same value in a specific column (e.g., one streak per user).
- **IntegrityError:** This is thrown when a database operation violates these rules. 
- **Real-world example:** In our gamification system, we use a unique constraint on `user_id` in the `Streak` table to prevent duplicate streak entries which would break the logic.
