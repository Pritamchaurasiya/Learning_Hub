SECTION 1: PROJECT OVERVIEW
================================================================================

WHAT IS NLP STUDIO?
--------------------
NLP Studio is a client-side Natural Language Processing web application.
It runs ENTIRELY in your browser — no server, no API keys, no backend.

WHAT DOES IT DO?
  - Sentiment Analysis: Is this text positive, negative, or neutral?
  - Named Entity Recognition: Find people, places, organizations in text
  - Summarization: Condense long text to key sentences
  - Translation: Convert text between 28 languages
  - Intent Detection: What is the user trying to do?

WHY BUILD THIS?
  - Learn NLP without needing a Python/ML backend
  - Understand modern web architecture patterns
  - Practice production-grade code quality
  - Demonstrate all major frontend engineering concepts

REAL-WORLD ANALOGY:
  Think of NLP Studio like a Swiss Army knife for text. A physical knife has
  different tools (blade, screwdriver, scissors). Similarly, our app has
  different "engines" (sentiment, NER, summarizer) that each do one job well.

SECTION 2: ARCHITECTURE — How a Modern SPA Works
================================================================================

WHAT IS A SINGLE PAGE APPLICATION (SPA)?
-----------------------------------------
Traditional websites: Every click loads a NEW HTML page from the server.
SPA: ONE HTML page loads ONCE, then JavaScript updates the content dynamically.

Think of it like:
  - Traditional: Changing TV channels (each channel = new page)
  - SPA: Same TV, but the picture changes (same page, different content)

OUR ARCHITECTURE:

  ┌──────────────────────────────────────────────────────────┐
  │                    index.html (Shell)                      │
  │  ┌─────────┐  ┌──────────────────────┐  ┌──────────────┐ │
  │  │ Sidebar  │  │    Main Content      │  │ Toast System │ │
  │  │   Nav    │  │  (Pages rendered     │  │ (Notifs)     │ │
  │  │         │  │   dynamically by JS)  │  │              │ │
  │  └─────────┘  └──────────────────────┘  └──────────────┘ │
  └──────────────────────────────────────────────────────────┘

  src/
  ├── core/           ← Foundation (store, events, security, logger)
  ├── nlp/            ← NLP Engines (sentiment, ner, summarizer, etc.)
  ├── styles/         ← CSS Design System
  └── main.js         ← App shell, routing, UI rendering

HASH-BASED ROUTING:
  URL: http://localhost/#dashboard  → Shows Dashboard page
  URL: http://localhost/#analyzer   → Shows Analyzer page
  URL: http://localhost/#settings   → Shows Settings page

  The "#" part is called the "hash". JavaScript listens for hash changes
  and renders the appropriate page. The browser NEVER actually navigates away.

  window.addEventListener('hashchange', () => {
    const page = window.location.hash.slice(1); // Remove the "#"
    renderPage(page);
  });

WHY THIS MATTERS:
  - Fast: No page reloads, instant navigation
  - Smooth: Animations and transitions between pages
  - Offline-capable: Everything runs client-side

COMMON MISTAKE:
  Beginners put ALL code in one file. We split into modules because:
  - Each file has ONE responsibility (Single Responsibility Principle)
  - Easier to test individual pieces
  - Easier for teams to work in parallel
  - Smaller files = easier to understand

SECTION 3: MODULE SYSTEM — ES Modules
================================================================================

WHAT ARE ES MODULES?
---------------------
Before modules, all JavaScript shared one global scope. This caused:
  - Name collisions (two libraries defining same variable names)
  - Hard to track dependencies
  - Can't tree-shake (remove unused code)

ES Modules solve this with `import` and `export`:

  // security.js — EXPORTS functions
  export function escapeHtml(str) { ... }
  export function sanitizeInput(input) { ... }

  // main.js — IMPORTS what it needs
  import { escapeHtml, sanitizeInput } from './core/security.js';

KEY CONCEPTS:
  - Named exports: export function foo() {}   → import { foo } from ...
  - Default export: export default class {}    → import MyClass from ...
  - Re-exports: export { foo } from './bar.js'

REAL-WORLD ANALOGY:
  Think of modules like rooms in a house:
  - Kitchen (security.js) has specific tools (knives, oven)
  - Living room (main.js) can BORROW tools from the kitchen
  - But it can't access the kitchen's private storage (unexported variables)

WHEN TO USE:
  - Always. There's no reason to NOT use modules in modern JS.
  
WHAT GOES WRONG IF MISUSED:
  - Circular dependencies: A imports B, B imports A → crashes
  - God modules: One module does everything → unmaintainable
  - Over-splitting: 100+ tiny files → navigation nightmare

SECTION 4: STATE MANAGEMENT — Reactive Store
================================================================================

THE PROBLEM:
  In a complex app, multiple UI components need the same data:
  - Dashboard needs analysis count
  - History page needs analysis list
  - Settings page needs configuration values

  Without centralized state, data gets scattered and inconsistent.

THE SOLUTION: CENTRALIZED STORE
  One "single source of truth" that all components read from and write to.

  // Create store
  const store = createStore({
    app: { currentPage: 'dashboard', loading: false },
    settings: { theme: 'dark', language: 'en' }
  });

  // Read state (path notation)
  store.getState('app.currentPage');  // 'dashboard'
  store.getState('settings.theme');   // 'dark'

  // Write state
  store.setState('app.loading', true);

  // Listen for changes
  store.subscribe('app.loading', (newValue) => {
    // Update UI when loading state changes
    showSpinner(newValue);
  });

HOW IT WORKS INTERNALLY:
  1. State is a plain JavaScript object
  2. Proxy/wrapper intercepts all reads and writes
  3. On write → notifies all matching subscribers
  4. Deep cloning prevents external mutation

  Path notation: 'a.b.c' → navigates state.a.b.c
  This is like a file system: /users/john/documents

PERSISTENCE (localStorage):
  Some state should survive page reloads:
  - Settings (user preferences)
  - History (previous analyses)

  We automatically save these to localStorage:
    localStorage.setItem('nlp-studio:settings', JSON.stringify(settings));

  On app start, we load saved state:
    const saved = JSON.parse(localStorage.getItem('nlp-studio:settings'));

COMMON MISTAKE:
  - Mutating state directly instead of using setState()
  - Forgetting to unsubscribe (causes memory leaks)
  - Storing too much data (localStorage has ~5MB limit)

MENTAL MODEL:
  Think of the store like a whiteboard in an office:
  - Anyone can READ the whiteboard
  - Only the STORE can WRITE to it (via setState)
  - People can SUBSCRIBE to be notified when something changes

SECTION 5: EVENT-DRIVEN ARCHITECTURE — Pub/Sub
================================================================================

WHAT IS PUB/SUB?
  Publisher/Subscriber pattern — components communicate WITHOUT knowing
  about each other.

  // Publisher (doesn't know who's listening)
  bus.emit('toast', { message: 'Analysis complete!', type: 'success' });

  // Subscriber (doesn't know who sends)
  bus.on('toast', ({ message, type }) => {
    showToast(message, type);
  });

WHY USE IT?
  Without pub/sub, component A must directly call component B:
    function analyzeText() {
      result = doAnalysis();
      toastSystem.show('Done!');     // A KNOWS about toast system
      historyTracker.add(result);    // A KNOWS about history
      logger.info('Complete');        // A KNOWS about logger
    }

  With pub/sub, A just announces what happened:
    function analyzeText() {
      result = doAnalysis();
      bus.emit('analysis:complete', result);  // A doesn't care who listens
    }

  Benefits:
  - Loose coupling: Components don't depend on each other
  - Easy to add: New feature? Just subscribe to existing events
  - Easy to remove: Delete subscriber without touching publisher

REAL-WORLD ANALOGY:
  Pub/Sub is like a newspaper:
  - The newspaper (publisher) prints news
  - Subscribers get the paper delivered
  - The newspaper doesn't know/care how many subscribers there are
  - Subscribers don't know how the newspaper operates internally

SECTION 6: SECURITY — Defense-in-Depth
================================================================================

XSS (Cross-Site Scripting):
----------------------------
THE ATTACK:
  User enters: <script>document.cookie</script>
  If we render this as HTML, the browser EXECUTES the script!
  The attacker can steal cookies, session tokens, and user data.

THE DEFENSE:
  HTML Entity Escaping — converts dangerous characters:
    < → &lt;
    > → &gt;
    " → &quot;
    ' → &#x27;
    & → &amp;

  function escapeHtml(str) {
    return str.replace(/[&<>"'\/`]/g, char => ENTITY_MAP[char]);
  }

  Now <script> becomes &lt;script&gt; — displayed as TEXT, not executed.

INPUT SANITIZATION:
  Goes beyond escaping — completely REMOVES dangerous patterns:
    - HTML tags: <b>bold</b> → bold
    - javascript: URLs: javascript:alert(1) → alert(1)
    - Event handlers: onerror=steal() → steal()

PROTOTYPE POLLUTION:
  JSON.parse('{"__proto__":{"admin":true}}')
  This pollutes Object.prototype, making ALL objects have admin=true!
  
  Defense: Strip __proto__, constructor, prototype keys recursively.

RATE LIMITING:
  Without limits, an attacker can send 1000 API requests per second.
  We use a sliding window limiter:
    const limiter = createRateLimiter(30, 60000); // 30 requests per minute
    if (!limiter().allowed) throw new Error('Rate limited');

CSRF (Cross-Site Request Forgery):
  Attacker tricks your browser into making requests to another site.
  Defense: Generate unique tokens per session:
    const token = generateCSRFToken(); // 64-char hex

CSP (Content Security Policy):
  HTTP header that tells the browser WHAT sources are allowed:
    script-src 'self'           → Only scripts from our domain
    connect-src 'self' https:// → Only fetch to our domain + HTTPS
    style-src 'self'            → Only stylesheets from our domain

  This PREVENTS loading malicious scripts from attacker domains.

SECTION 7: NLP FUNDAMENTALS
================================================================================

WHAT IS NLP?
  Natural Language Processing = teaching computers to understand human language.
  It's the technology behind: Siri, Google Translate, ChatGPT, spam filters.

7A. SENTIMENT ANALYSIS — Is It Positive or Negative?
-----------------------------------------------------
HOW IT WORKS (AFINN Lexicon):
  A dictionary maps words to sentiment scores:
    "amazing" → +4
    "terrible" → -4
    "good" → +3
    "bad" → -3
    "the" → 0 (neutral, not in dictionary)

  Algorithm:
  1. Split text into words (tokenize)
  2. Look up each word in the AFINN dictionary
  3. Handle negation: "not good" → flip sign of "good" (-3)
  4. Handle intensifiers: "very good" → boost score (3 × 1.5 = 4.5)
  5. Handle emojis: 😊 → +2, 😢 → -2
  6. Sum all scores → overall sentiment

  Example: "This movie is not very good but the acting is amazing"
    "not" → triggers negation
    "very" → triggers intensifier
    "good" → +3, but negated and intensified → -4.5
    "amazing" → +4
    Total: -4.5 + 4 = -0.5 → Slightly negative

CONFIDENCE CALCULATION:
  confidence = (found words / total words) × (1 - std_dev / max_range)
  If we only found 2 sentiment words in 100 words, confidence is low.

7B. NAMED ENTITY RECOGNITION (NER)
------------------------------------
WHAT: Finding and classifying proper nouns in text.
  "John Smith works at Google in New York" →
    [John Smith: PERSON] [Google: ORGANIZATION] [New York: PLACE]

HOW (using compromise.js):
  1. Parse text into a document model (compromise)
  2. Use linguistic rules to identify:
     - People: Capitalized names + title patterns (Mr., Dr.)
     - Places: Known geographic terms + patterns
     - Organizations: Company name patterns
  3. For dates: Regex patterns (January 15th, tomorrow, next week)
  4. Deduplicate (case-insensitive)
  5. Calculate positions in original text
  6. Generate annotated HTML with colored spans

WHY NOT ML?
  Full ML-based NER (like spaCy) requires 200MB+ models.
  compromise.js uses linguistic rules (50KB) — perfect for client-side.

7C. TEXT SUMMARIZATION (TF-IDF Extractive)
-------------------------------------------
WHAT: Reduce a long document to its most important sentences.

TF-IDF explained:
  TF (Term Frequency): How often does this word appear in this sentence?
  IDF (Inverse Document Frequency): How rare is this word across all sentences?
  TF-IDF = TF × IDF

  Words that appear in EVERY sentence (like "the") get LOW scores.
  Words that appear in FEW sentences get HIGH scores.

Algorithm:
  1. Split text into sentences
  2. Calculate TF-IDF for each word in each sentence
  3. Score each sentence = sum of its word TF-IDF scores
  4. Add position bias: First sentences are often introductory
  5. Add length bias: Very short sentences are less informative
  6. Select top N sentences (based on ratio, e.g., 30%)
  7. Return sentences in original order

7E. INTENT DETECTION (Pattern Matching)
-----------------------------------------
WHAT: Classify what the user wants to do.
  "What is machine learning?" → QUESTION intent
  "Book a flight to Paris" → BOOKING intent
  "I'm so frustrated!" → COMPLAINT intent

HOW:
  Each intent has keyword patterns:
    QUESTION: ["what", "how", "why", "when", "?"]
    BOOKING: ["book", "reserve", "schedule", "appointment"]
    COMPLAINT: ["frustrated", "angry", "terrible", "worst"]

  Algorithm:
  1. Tokenize and lowercase input
  2. Calculate match score for each intent
  3. Normalize to confidence percentages
  4. Return sorted by confidence

SECTION 8: CSS DESIGN SYSTEM — Glassmorphism
================================================================================

WHAT IS A DESIGN SYSTEM?
  A set of reusable rules (colors, spacing, typography) that ensure
  consistency across the entire application.

CSS CUSTOM PROPERTIES (Variables):
  :root {
    --color-accent-primary: #6c63ff;
    --space-4: 1rem;
    --radius-md: 0.75rem;
  }

  .button {
    background: var(--color-accent-primary);
    padding: var(--space-4);
    border-radius: var(--radius-md);
  }

  Benefits: Change ONE variable → updates EVERYWHERE.

GLASSMORPHISM:
  A design trend using:
  1. Frosted glass background: rgba(255, 255, 255, 0.05)
  2. Blur effect: backdrop-filter: blur(20px)
  3. Subtle border: 1px solid rgba(255, 255, 255, 0.1)
  4. Shadow: box-shadow with transparency

  .card {
    background: var(--gradient-glass);     /* Semi-transparent */
    backdrop-filter: blur(20px);           /* Frosted glass */
    border: 1px solid rgba(255,255,255,0.08);
    box-shadow: 0 8px 32px rgba(0,0,0,0.3);
  }

RESPONSIVE DESIGN:
  @media (max-width: 768px) {
    .sidebar { transform: translateX(-100%); }  /* Hidden on mobile */
    .stats-grid { grid-template-columns: 1fr; } /* Stack vertically */
  }

ACCESSIBILITY:
  @media (prefers-reduced-motion: reduce) {
    * { animation-duration: 0.01ms !important; }  /* Disable animations */
  }

  @media (prefers-contrast: high) {
    :root { --color-text-primary: #ffffff; }  /* Max contrast */
  }

SECTION 9: TESTING — Vitest Unit Testing
================================================================================

WHY TEST?
  - Catch bugs before users do
  - Confidence to refactor code
  - Documentation of expected behavior
  - Prevent regressions (breaking existing features)

VITEST (our test framework):
  import { describe, it, expect } from 'vitest';

  describe('escapeHtml', () => {
    it('escapes angle brackets', () => {
      expect(escapeHtml('<script>')).toBe('&lt;script&gt;');
    });

    it('handles empty string', () => {
      expect(escapeHtml('')).toBe('');
    });

    it('handles null gracefully', () => {
      expect(escapeHtml(null)).toBe('');
    });
  });

TEST STRUCTURE:
  describe() → Group related tests
  it()       → One specific behavior
  expect()   → Assert expected outcome

EDGE CASES TO ALWAYS TEST:
  1. Empty input (null, undefined, '', [], {})
  2. Very large input (100,000 characters)
  3. Special characters (<, >, ", ', &)
  4. Unicode/emoji (🎉, 中文, العربية)
  5. Boundary values (0, -1, MAX_INT)

SECTION 10: PERFORMANCE OPTIMIZATION
================================================================================

LAZY LOADING:
  Don't load the NLP engine until the user needs it:
    async function loadNLP() {
      if (!nlp) {
        const module = await import('compromise'); // Only loads when called
        nlp = module.default;
      }
      return nlp;
    }

BUNDLE SPLITTING (Vite):
  vite.config.js → manualChunks splits code into separate files:
    vendor: ['compromise', 'dompurify']  → Cached separately
    nlp:    ['pipeline.js']              → Only loaded on Analyzer page

CACHING:
  Translation cache: Map with LRU eviction (max 200 entries)
  Browser cache: Static assets cached by HTTP headers

SECTION 11: ACCESSIBILITY (WCAG 2.1 AA)
================================================================================

KEY REQUIREMENTS:
  1. Color contrast ratio ≥ 4.5:1 for normal text
  2. All interactive elements keyboard-accessible
  3. Screen reader compatibility (ARIA attributes)
  4. No information conveyed by color alone
  5. Reduced motion support

ARIA ATTRIBUTES WE USE:
  role="navigation"      → Tells screen readers this is a nav
  role="tablist"          → Tab container
  role="tab"             → Individual tab
  aria-selected="true"   → Currently active tab
  aria-live="polite"     → Auto-announce content changes
  aria-label="..."       → Describes element purpose

KEYBOARD NAVIGATION:
  Tab         → Move between interactive elements
  Enter/Space → Activate buttons
  ArrowKeys   → Navigate between tabs
  Escape      → Close modals/sidebar

SECTION 12: CYBERSECURITY DEEP DIVE
================================================================================

ATTACK: XSS (Cross-Site Scripting)
  Vector: User input rendered as HTML
  Impact: Cookie theft, session hijacking, keylogging
  Defense: escapeHtml(), CSP headers, DOMPurify
  
ATTACK: Prototype Pollution
  Vector: JSON.parse with __proto__ keys
  Impact: Modify Object.prototype globally
  Defense: stripDangerousKeys() in safeJsonParse()

ATTACK: ReDoS (Regular Expression Denial of Service)
  Vector: Malicious input causing regex backtracking
  Impact: CPU freeze, application hang
  Defense: Input length limits (MAX_SANITIZE_LENGTH = 100K)

ATTACK: CSRF (Cross-Site Request Forgery)
  Vector: Trick browser into making authenticated requests
  Impact: Unauthorized actions on behalf of user
  Defense: generateCSRFToken() per session

ATTACK: Clickjacking
  Vector: Embed your site in an iframe on attacker's site
  Defense: X-Frame-Options: DENY or CSP frame-ancestors 'none'

KEY PRINCIPLE — DEFENSE IN DEPTH:
  Never rely on just ONE defense. Layer multiple protections:
  1. Input validation (front-end)
  2. Input sanitization (escapeHtml)
  3. CSP headers (browser enforcement)
  4. Rate limiting (abuse prevention)
  5. Error handling (don't leak info)

SECTION 13: BUILD TOOLS — Vite
================================================================================

WHAT IS VITE?
  A build tool that:
  1. Development: Uses native ES modules for instant Hot Module Replacement
  2. Production: Bundles with Rollup for optimized output

WHY VITE?
  Traditional (Webpack): Bundles EVERYTHING before you see anything (slow)
  Vite: Serves files directly, only transforms what's requested (instant)

HOW HMR WORKS:
  1. You edit a file
  2. Vite detects the change
  3. Only that module is re-sent to the browser
  4. Browser updates without full page reload

SECTION 14: ERROR HANDLING
================================================================================

GLOBAL ERROR HANDLER:
  window.addEventListener('error', handleError);
  window.addEventListener('unhandledrejection', handlePromiseError);

  This catches ANY uncaught error in the entire application.

ERROR BOUNDARY PATTERN:
  try {
    renderPage();
  } catch (err) {
    showErrorFallback(err.message);
  }

  If a page crashes, show a friendly error message instead of a white screen.

SAFE FUNCTION WRAPPER:
  function safe(fn) {
    return (...args) => {
      try { return fn(...args); }
      catch (err) { logger.error(err); return null; }
    };
  }

SECTION 15: MENTAL MODELS — Think Like a Senior Engineer
================================================================================

1. SEPARATION OF CONCERNS
   Each module should have ONE job. security.js doesn't do NLP.
   NLP engines don't do UI rendering.

2. FAIL GRACEFULLY
   Never show a blank page. Always have a fallback:
   - API fails? Show cached data.
   - NLP engine crashes? Show error message, not white screen.

3. TRUST NOTHING FROM THE USER
   All user input is potentially malicious.
   Always sanitize, validate, and escape.

4. PERFORMANCE BUDGET
   Set limits: "Page must load in < 2 seconds"
   Measure: processingTime in every analysis result
   Optimize: Lazy load, cache, debounce

5. TEST THE EDGES
   The happy path always works. Test:
   - What if input is empty?
   - What if the API is down?
   - What if the user clicks 100 times rapidly?
   - What if text is in Chinese? Arabic? Emoji?

6. DRY (Don't Repeat Yourself)
   If you write the same code twice, extract it into a function.
   Our `addEntity()` helper replaces 5 copy-pasted blocks.

7. YAGNI (You Aren't Gonna Need It)
   Don't build features nobody asked for.
   Add complexity only when justified by real requirements.

8. MAKE IT WORK → MAKE IT RIGHT → MAKE IT FAST
   First: Get it working (quick & dirty is OK)
   Then: Refactor for quality (clean code, tests)
   Finally: Optimize for performance (only where needed)
