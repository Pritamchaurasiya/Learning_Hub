# Learning Hub: Advanced Flutter Engineering Guide

## 1. Architectural Foundation: Feature-First & Clean Architecture

### What is it?
We structured this project using a **Feature-First** approach combined with **Clean Architecture** principles.
- **Feature-First**: Code is organized by what it *does* (e.g., `features/courses`, `features/auth`), not just what it *is* (e.g., `controllers`, `views`).
- **Clean Architecture**: Separates code into three layers:
    1.  **Domain**: Entities and business logic (pure Dart, no Flutter).
    2.  **Data**: Repositories, API clients, and data sources.
    3.  **Presentation**: Widgets, Controllers (Riverpod providers).

### Why is it needed?
As apps grow, "Layer-First" (putting all controllers in one folder) becomes unmanageable. You lose context. Feature-first keeps related code together. Clean Architecture ensures your business logic isn't coupled to the UI, making it testable and reusable.

### Common Mistakes
- **Leaking Logic to UI**: Putting API calls directly in `build()` methods.
- **Tight Coupling**: Making Repositories depend on Controllers. The dependency flow should be: UI -> Controller -> UseCase/Repository -> Data Source.

## 2. Functional Error Handling with `fpdart`

### What is it?
Instead of throwing exceptions (`try-catch`), we use the `Either<L, R>` type from `fpdart`.
- **Left (L)**: Represents failure (e.g., `NetworkFailure`).
- **Right (R)**: Represents success (e.g., `List<Course>`).

### How it works
```dart
Future<Either<Failure, Course>> getCourse() async {
  try {
    return Right(await api.fetch());
  } catch (e) {
    return Left(ServerFailure(e.toString()));
  }
}
```
The caller *forcefully* handles both cases using `.fold()`:
```dart
result.fold(
  (failure) => showError(failure),
  (course) => showCourse(course),
);
```

### Why is it superior?
- **Type Safety**: You can't accidentally ignore an error. The compiler forces you to handle the `Left` case.
- **Composability**: You can chain operations (map, flatMap) without nesting try-catch blocks.
- **Predictability**: Functions signature tells the whole truth. `Future<Course>` relies on implicit exceptions; `Future<Either<Failure, Course>>` makes failure explicit.

## 3. Network Resilience & Offline-First

### The Problem
Mobile networks are flaky. A simple `http.get` is naive.
We implemented two layers of defense:
1.  **Retry Policy**: Using `dio_smart_retry`, we automatically retry failed requests (e.g., 3 times with exponential backoff). This handles transient glitches like switching cell towers.
2.  **Connectivity Awareness**: We injected `NetworkInfo` into our Repository. Before even attempting a request, we check `internet_connection_checker`. If offline, we return cached data immediately or a friendly `NetworkFailure`.

### Security & Performance
- **Throttling/Debouncing**: Retrying too fast can DDoS your own server. Exponential backoff (1s, 2s, 4s) prevents this.
- **Caching**: We use a simple in-memory cache time-stamped for validity (5 mins). In a real production app, use a persistent DB (Hive/Isar) to allow "Offline Mode" across app restarts.

## 4. Performance: The Cost of Glassmorphism

### The Insight
We optimized `GlassContainer`. The `BackdropFilter` widget is expensive because it requires the GPU to read from the texture behind it, apply a convolution kernel (blur), and write it back.
Doing this for *every item in a list* (e.g., Course Curriculum) causes massive frame drops (jank) during scrolling.

### The Fix
We made the blur conditional:
```dart
if (blur > 0) {
  return BackdropFilter(...);
} else {
  return Container(...); // much cheaper
}
```
For the list items, we set `blur: 0`. The UI still looks consistent (translucent), but scrolling is 60fps smooth.

## 5. Security: Defensive Coding

### Input Validation
Never trust backend data types explicitly.
```dart
// Bad
final names = json['names'] as List<String>; 

// Good (Defensive)
final list = json['names'];
final names = (list is List) ? list.map((e) => e.toString()).toList() : [];
```
Our `_extractResults` helper in `CourseRepository` prevents crashes when the API returns unexpected structures (e.g., empty list instead of map).

### Secure Storage
Configuration secrets (like API keys) should ideally be injected at build time, but if needed at runtime, use `flutter_secure_storage` instead of `SharedPreferences`. SharedPrefs is plain text XML/JSON on disk!
