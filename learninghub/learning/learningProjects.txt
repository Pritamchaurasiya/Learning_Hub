## 1. Variable Shadowing & Closures

### What Is It?
Variable shadowing occurs when a variable declared in an inner scope has the same name as a variable in an outer scope. The inner variable "shadows" the outer one, making the outer one inaccessible.

### Why It's Dangerous
```typescript
// ❌ BAD: This caused a real bug in QuizPage.tsx
const { progress } = useStore()  // Zustand store progress (xp, streak, level)

// ... 300 lines later ...

const progress = ((currentQuestionIndex + 1) / quiz.questions.length) * 100  // quiz progress %

// When handleSubmit references `progress`, it gets the QUIZ percentage,
// not the STORE progress — leading to incorrect XP calculations
```

### The Fix
```typescript
// ✅ GOOD: Unique, descriptive variable names eliminate ambiguity
const { progress } = useStore()  // store progress

const quizProgress = ((currentQuestionIndex + 1) / quiz.questions.length) * 100
```

### Mental Model
Think of variable names as mailing addresses. If two houses on the same street have the same number, mail goes to the wrong place. Always use unique, descriptive names.

### Common Mistakes
- Using `data` as a variable name in every component
- Using `result` for both API responses and computed values
- Using `error` for both caught exceptions and validation errors

## 2. React useEffect Dependency Arrays

### What Is It?
The dependency array in `useEffect` tells React which values to watch. When any dependency changes, the effect re-runs.

### Why Missing Dependencies Cause Bugs
```typescript
// ❌ BAD: handleSubmit is NOT in the dependency array
useEffect(() => {
  const timer = setInterval(() => {
    setTimeRemaining((prev) => {
      if (prev <= 1) {
        handleSubmit()  // This captures the INITIAL handleSubmit closure
        return 0
      }
      return prev - 1
    })
  }, 1000)
  return () => clearInterval(timer)
}, [quiz, result, timeRemaining])  // Missing: handleSubmit
```

When the timer fires after 5 minutes, `handleSubmit` still has the answers from when the effect was created, not the current answers.

### The Fix
```typescript
// ✅ GOOD: Wrap in useCallback and include in deps
const handleSubmitCallback = useCallback(() => {
  handleSubmit()
}, [quiz, answers, timeRemaining])

useEffect(() => {
  // ... timer code calling handleSubmitCallback ...
}, [quiz, result, timeRemaining, handleSubmitCallback])
```

### Mental Model
Think of useEffect as a photograph. The dependency array decides when to take a new photo. If a value changes but isn't in the array, the effect keeps looking at the old photo.

## 3. API Error Handling & Token Refresh

### What Is It?
Token refresh is when an expired access token is automatically replaced with a new one using a refresh token, without requiring the user to log in again.

### The Race Condition We Fixed
```typescript
// ❌ BAD: After refresh, the NEW response is never checked
if (response.status === 401 && token) {
  const newToken = await refreshToken()
  headers.set('Authorization', `Bearer ${newToken}`)
  response = await fetch(url, { headers })
  // Falls through to error handling below, which checks the OLD response state
}

// This error handling runs on the REFRESHED response,
// but was written assuming it's the original failed response
const errorData = await response.json()  // Could fail if refresh succeeded!
```

### The Fix
```typescript
// ✅ GOOD: Check the refreshed response immediately
if (response.status === 401 && token) {
  const newToken = await refreshToken()
  headers.set('Authorization', `Bearer ${newToken}`)
  response = await fetch(url, { headers })
  
  if (response.ok) {
    return response.json()  // Early return on success!
  }
}
// Error handling only runs if refresh also failed
```

### Common Mistakes
- Not checking `response.ok` after every fetch
- Allowing `null` responses to propagate (`response?.json() ?? {}` hides errors)
- Using `window.location.href` for redirects (loses all app state)

## 4. Content Security Policy (CSP)

### What Is It?
CSP is an HTTP header that tells browsers which resources are allowed to load. It prevents XSS attacks by blocking unauthorized scripts, styles, and connections.

### Why Our CSP Blocked Dev Mode
```html
<!-- ❌ BAD: Missing ws: in connect-src -->
<meta http-equiv="Content-Security-Policy" 
  content="connect-src 'self' https://api.example.com" />

<!-- Vite uses WebSockets (ws://) for Hot Module Replacement -->
<!-- The CSP blocks WebSocket connections → HMR doesn't work in dev -->
```

### The Fix
```html
<!-- ✅ GOOD: Allow WebSocket connections for dev tools -->
<meta http-equiv="Content-Security-Policy" 
  content="connect-src 'self' https://api.example.com ws: wss:" />
```

### Security Note
In production, you'd typically:
1. Remove `ws: wss:` from CSP (only needed for dev)
2. Use a build-time plugin to inject different CSP headers per environment
3. Use HTTP headers instead of `<meta>` tags (more secure, can't be tampered with)

## 5. Accessibility (WCAG 2.1 AA)

### Toggle Switches Need Labels
```typescript
// ❌ BAD: Screen readers say "switch, unchecked" with no context
<button role="switch" aria-checked={value} />

// ✅ GOOD: Screen readers say "Toggle daily reminder notifications, switch, unchecked"
<button 
  role="switch" 
  aria-checked={value}
  aria-label="Toggle daily reminder notifications"
/>
```

### Live Regions
```typescript
// ❌ BAD: Duplicate announcement — screen reader says count twice
<p>
  <span className="sr-only">5 courses found</span>  {/* Announced */}
  <span aria-hidden="true">5 courses found</span>    {/* Hidden but visible */}
</p>

// ✅ GOOD: Single element handles both visual and screen reader
<p aria-live="polite" aria-atomic="true">
  5 courses found
</p>
```

### Text Direction
```html
<!-- ❌ Missing: Screen readers may guess text direction wrong -->
<html lang="en">

<!-- ✅ Explicit direction for assistive technology -->
<html lang="en" dir="ltr">
```

## 6. TypeScript Type Safety

### Why `any` Is Dangerous
```typescript
// ❌ BAD: No type checking — bugs slip through
interface AuthState {
  user: any | null  // Could be string, number, object... anything
}

// This compiles but crashes at runtime:
const username = auth.user.username.toUpperCase()  // user might not have username!
```

### The Fix
```typescript
// ✅ GOOD: Compiler catches mistakes before runtime
interface User {
  id: string | number
  email: string
  username?: string  // Optional — forces null checks
  xp: number
  level: number
  streak: number
  lastActive: string
}

interface AuthState {
  user: User | null
}

// Now TypeScript forces you to check:
const username = auth.user?.username?.toUpperCase() ?? 'Unknown'
```

## 7. React Performance Patterns

### Lazy Loading
```typescript
// ❌ BAD: All pages loaded upfront, even if user never visits them
import { AuthPage } from './pages/AuthPage'
import HomePage from './pages/HomePage'

// ✅ GOOD: Pages load only when navigated to
const AuthPage = lazy(() => import('./pages/AuthPage'))
const HomePage = lazy(() => import('./pages/HomePage'))
```

### Code Splitting Results
Our build shows the impact:
- `AuthPage-B9M-QTSX.js`: 2.57 KB (only loaded on /auth)
- `QuizPage-LjnDI5Dt.js`: 10.07 KB (only loaded on /quiz)
- `index-DfS5DoFB.js`: 99.95 KB (shared core, always loaded)

## 8. Security Best Practices

### localStorage Token Storage (Known Risk)
```
⚠️ RISK LEVEL: MEDIUM
⚠️ ATTACK: XSS (Cross-Site Scripting)

HOW IT WORKS:
1. Attacker injects script via XSS vulnerability
2. Script reads localStorage.getItem('token')
3. Token is sent to attacker's server
4. Attacker impersonates the user

MITIGATION:
- Use httpOnly cookies (browser prevents JS access)
- Implement token rotation (short-lived access tokens)
- Add CSP headers to prevent XSS
- Sanitize all user input (we use DOMPurify)
```

### DOMPurify Configuration
```typescript
// Our markdown.ts adds buttons to allowed tags for copy functionality
const purifyConfig = {
  ADD_ATTR: ['data-code', 'class', 'title', 'type'],
  ADD_TAGS: ['button'],
}

// ⚠️ This is safe ONLY because:
// 1. We use delegated event handlers (no onclick in HTML)
// 2. DOMPurify strips all event handlers by default
// 3. data-code is used only for copy-to-clipboard (read-only)
```

## 9. Code Architecture Anti-patterns

### Double Padding
```typescript
// ❌ BAD: Layout adds padding, then page adds more
// Layout.tsx:
<main className="p-4 md:p-6 lg:p-8">  {/* Outer padding */}
  <Outlet />
</main>

// ProfilePage.tsx:
<div className="max-w-4xl mx-auto p-4 md:p-6">  {/* Inner padding = double! */}

// ✅ GOOD: Only Layout handles padding, pages handle max-width
// ProfilePage.tsx:
<div className="max-w-4xl mx-auto space-y-6">  {/* No padding */}
```

### Redundant Auth Guards
```typescript
// ❌ BAD: Both ProtectedRoute AND the page check auth
// App.tsx:
<Route element={<ProtectedRoute><ProfilePage /></ProtectedRoute>} />

// ProfilePage.tsx:
useEffect(() => {
  if (!auth.isAuthenticated) navigate('/auth')  // Dead code!
}, [auth])

// ✅ GOOD: Single source of truth — ProtectedRoute handles it
```

### Duplicate Logic
```typescript
// ❌ BAD: Three functions that do the same thing
addBookmark(courseId)    // POST /progress/bookmark
removeBookmark(courseId) // POST /progress/bookmark  ← Same endpoint!
toggleBookmark(courseId) // POST /progress/bookmark  ← Same endpoint!

// ✅ GOOD: Single source + convenience wrappers
toggleBookmark(courseId) // The real implementation
addBookmark: (id) => get().toggleBookmark(id)     // Thin wrapper
removeBookmark: (id) => get().toggleBookmark(id)   // Thin wrapper
```

## 10. Production Build Optimization

### Bundle Analysis
```
Total build: ~1.5 MB (508 KB gzipped)

  markdown-D-Mted9l.js  998 KB  ← highlight.js (consider lazy-loading)

  index-DfS5DoFB.js      99 KB  ← App core

All page chunks: < 34 KB each ✅ (lazy loaded)
```

### Future Optimization Ideas
1. **Lazy-load highlight.js languages** — Only load syntax highlighting for languages actually used
2. **Use `React.memo()`** for expensive list items (course cards, problem cards)
3. **Add `will-change: transform`** to animated elements for GPU acceleration
4. **Implement virtual scrolling** for large course/problem lists (200+ items)

## Quick Reference: Bug Categories

| Category | Risk | Detection | Prevention |
|----------|------|-----------|------------|
| Variable Shadowing | High | ESLint `no-shadow` rule | Descriptive naming |
| Stale Closures | High | React exhaustive-deps lint | useCallback + proper deps |
| Race Conditions | Critical | Code review, integration tests | Check response.ok, mutex locks |
| CSP Misconfiguration | Medium | Browser console errors | Environment-specific CSP |
| Missing ARIA | Medium | axe-core, screen reader testing | Component-level a11y checklist |
| Type Safety | Medium | `strict: true` in tsconfig | Ban `any` via ESLint |
| Double Padding | Low | Visual inspection | Single responsibility for spacing |
| Duplicate Logic | Low | Code review | DRY principle, thin wrappers |
