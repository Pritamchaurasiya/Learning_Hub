## 1. Project Overview & System Thinking

## 2. Technical Stack: The Modern Foundation

### Frontend: React 18 & Vite
- **React 18:** We use the latest React features like **Concurrent Rendering**.
  - **useTransition:** Allows us to mark state updates as "non-urgent" so the UI remains responsive during heavy renders.
  - **useDeferredValue:** Helps defer rendering part of the UI that takes a long time to calculate (like search results).
- **Vite:** A lightning-fast build tool that uses native ES modules for instant hot module replacement (HMR).

### Styling: Vanilla CSS & Tailwind
- We prioritize clean, premium aesthetics using Tailwind CSS for utility-first styling combined with custom design tokens.

### State Management: Zustand
- **Why?** Redux is often too boilerplate-heavy. Zustand is minimalist, scalable, and easy to use with hooks.
- **Our Pattern:** We use slices (authSlice, progressSlice) to keep the store modular.

## 3. Deep Security: Defensive & Offensive Patterns

### Authentication: httpOnly Cookies vs. localStorage
- **Beginner Mistake:** Storing JWTs in `localStorage`. 
  - **Risk:** XSS attacks can steal the token using `document.cookie` or simple scripts.
- **Advanced Solution:** `httpOnly` Cookies.
  - **How it works:** The browser automatically includes the cookie in every request to the backend. JavaScript cannot access it.
  - **Why?** It virtually eliminates token theft via XSS.

### CSRF (Cross-Site Request Forgery)
- **The Attack:** An attacker tricks a logged-in user into clicking a link that performs an action on the backend.
- **The Defense:** We use `SameSite=Strict` or `Lax` cookies and custom headers (like `X-Requested-With`) to ensure requests come from our trusted frontend.

## 4. Performance Advanced: Thinking Like a Senior Engineer

### Concurrent Rendering & UX
When a user types in a search bar, if the filtering logic is heavy, the typing lags. 
- **Without Concurrent React:** Typing -> Render Filtered List (Block UI) -> UI updates.
- **With Concurrent React:** Typing (High Priority) -> Start filtering in background (Low Priority) -> UI remains smooth.

### Optimized Images
- **loading="lazy":** Only loads images when they are about to enter the viewport.
- **decoding="async":** Decodes image data off the main thread to prevent frame drops.

---

## 5. Resilience: Building a Fault-Tolerant UI

### Error Boundaries & Widget Isolation
- **The Problem:** One small error in a notification widget crashes the whole app.
- **The Solution:** `WidgetBoundary`. 
  - We wrap individual components (widgets) in localized error boundaries. If one fails, the rest of the page stays alive.

---

## 6. Database Scalability: Prisma & PostgreSQL

### Connection Pooling
- **What?** Reusing existing database connections instead of opening/closing them for every request.
- **Why?** Opening a connection is expensive. In production (e.g., using Neon/Postgres), failing to pool can crash the server during traffic spikes.
- **The Fix:** `connection_limit=10` in your `DATABASE_URL`.

## 7. Real-World Engineering Practices

1. **DRY (Don't Repeat Yourself):** Centralize API calls in `utils/api.ts`.
2. **Clean Abstractions:** Use services (e.g., `certificateService.ts`) to handle business logic, keeping components clean.
3. **Audit Readiness:** Keep `.env.example` updated so other developers can set up the project in minutes.
4. **Testing:** Always run `vitest` to ensure your changes didn't break existing features.

---

## 8. What Goes Wrong if Misused?
- **Misusing `useMemo`:** Wrapping everything in `useMemo` can actually slow down the app due to memory overhead. Use it only for expensive calculations.
- **Missing CSRF Protection:** Your users' accounts could be compromised by malicious sites.
- **Poor Connection Management:** Your app will be fast for 1 user but crash for 100.
